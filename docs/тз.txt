## 3. ОСОБЕННОСТИ РЕАЛИЗАЦИИ

### Критически важные моменты

#### A. Виртуализация (TanStack Virtual)
```javascript
// Рендерим ТОЛЬКО видимые элементы
// В DOM всегда ~20-30 элементов, не 1,000,000
// Память: ~50KB вместо 50MB
```

#### B. Кэширование (TanStack Query)
```javascript
// После 100 скроллов вниз = 2000 элементов в кэше
// Скролл вверх = 0 запросов (мгновенно!)
// Поиск = отдельный кэш
// Перезагрузка = кэш очищается
```

#### C. Batch-запросы (производительность)
```javascript
// Вместо 1000 запросов → 1 запрос в секунду
// Экономия сервера и батареи пользователя
// Дедупликация: {select:1, deselect:1} → финальное состояние
```

#### D. Оптимистичные обновления (UX)
```javascript
// Клик → UI меняется мгновенно
// Сервер получит изменение через 0-1000мс
// При ошибке → откат с уведомлением
```


### Потоки данных (Data Flows)

#### A. Инициализация приложения (Initial Load)

```
[Пользователь открывает страницу]
    ↓
[Фронтенд монтируется]
    ↓
[Проверка localStorage] → есть сохраненное состояние? → [Восстановление fullOrder]
    ↓
[useInfiniteQuery('left-items')] → GET /api/left?offset=0&limit=20
[useInfiniteQuery('right-items')] → GET /api/right?offset=0&limit=20
    ↓
[Сервер возвращает первые 20 элементов для каждого списка]
    ↓
[TanStack Query кэширует данные]
    ↓
[Виртуализация рендерит видимые элементы]
    ↓
[UI готов к взаимодействию]
```

#### B. Скролл и пагинация (Infinite Scroll)

```
[Пользователь скроллит вниз]
    ↓
[Virtualizer определяет: осталось < 5 элементов до конца]
    ↓
[Триггер fetchNextPage()]
    ↓
[Проверка: hasNextPage && !isFetching]
    ↓
[Запрос: GET /api/left?offset={currentOffset}&limit=20]
    ↓
[Сервер возвращает следующую страницу]
    ↓
[TanStack Query добавляет страницу в кэш (pages[])]
    ↓
[allItems = pages.flatMap(page => page.items) обновляется]
    ↓
[Виртуализация получает обновленный массив]
    ↓
[Рендер новых элементов при скролле]
```

#### C. Поиск/фильтрация (Search)

```
[Пользователь печатает в поиск "123"]
    ↓
[useState обновляется]
    ↓
[useDebounce (300ms) ждет окончания ввода]
    ↓
[debouncedSearch обновляется]
    ↓
[queryKey = ['left-items', debouncedSearch] МЕНЯЕТСЯ]
    ↓
[TanStack Query определяет: новые данные нужны]
    ↓
[Старые страницы из кэша (с другим ключом) игнорируются]
    ↓
[Автоматический запрос: GET /api/left?search=123&offset=0&limit=20]
    ↓
[Сервер возвращает первые 20 результатов поиска]
    ↓
[Новый кэш для этого поиска создается]
    ↓
[UI обновляется]
```

#### D. Выбор элемента (Select) - Оптимистичное обновление

```
[Пользователь кликает "+" на элементе 42 в левом списке]
    ↓
[onSelect(42) вызывается]
    ↓
[queryClient.cancelQueries() отменяет текущие запросы]
    ↓
[Сохраняем предыдущее состояние для отката]
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Левый список]
    ↓
    ├─ Находим страницу с элементом 42
    ├─ Удаляем 42 из items этой страницы
    └─ Если страница стала пустой - удаляем страницу
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Правый список]
    ↓
    ├─ Добавляем 42 в конец последней страницы
    └─ fullOrder = [...fullOrder, 42]
    ↓
[UI МГНОВЕННО ОБНОВЛЯЕТСЯ (элемент переехал)]
    ↓
[Действие { type: 'select', id: 42 } добавляется в BATCH-очередь]
    ↓
[Таймер 1 сек: очередь отправляется на сервер]
    ↓
[POST /api/batch с массивом действий]
    ↓
[Сервер обрабатывает: globalState.selectedItemsOrder.push(42)]
    ↓
[200 OK]
    ↓
[Ничего не делаем - UI уже правильный]
```

#### E. Отмена выбора (Deselect) - Оптимистичное обновление

```
[Пользователь кликает "←" на элементе 42 в правом списке]
    ↓
[onDeselect(42) вызывается]
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Правый список]
    ↓
    ├─ Удаляем 42 из всех страниц right-items
    ├─ fullOrder = fullOrder.filter(id => id !== 42)
    └─ Пересчитываем пагинацию
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Левый список]
    ↓
    ├─ Находим правильную позицию для 42 (по ID)
    ├─ Вставляем в соответствующую страницу
    └─ Сортируем страницу
    ↓
[UI МГНОВЕННО ОБНОВЛЯЕТСЯ (элемент вернулся)]
    ↓
[Действие { type: 'deselect', id: 42 } в BATCH-очередь]
    ↓
[Отправка через 1 сек]
```

#### F. Drag & Drop сортировка (Reorder)

```
[Пользователь захватывает элемент за ⋮⋮ и перетаскивает]
    ↓
[dnd-kit: dragStart → activeId = 58]
    ↓
[Перетаскивание над элементом 23]
    ↓
[dnd-kit: dragEnd → activeId=58, overId=23]
    ↓
[Вычисляем новый порядок: arrayMove(fullOrder, oldIndex, newIndex)]
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Правый список]
    ↓
    ├─ fullOrder = новый порядок
    └─ Все страницы right-items перестраиваются согласно новому порядку
    ↓
[UI МГНОВЕННО ОБНОВЛЯЕТСЯ (элементы переставлены)]
    ↓
[Действие { type: 'reorder', order: newFullOrder } в BATCH-очередь]
    ↓
[Отправка через 1 сек]
```

#### G. Добавление нового элемента (Add New Item)

```
[Пользователь вводит "1000001" и нажимает "Добавить"]
    ↓
[Валидация: ID > 1,000,000 и не пустое]
    ↓
[ОПТИМИСТИЧНОЕ ОБНОВЛЕНИЕ: Левый список]
    ↓
    ├─ Добавляем { id: 1000001 } в конец последней страницы
    └─ totalCount увеличивается
    ↓
[UI показывает новый элемент]
    ↓
[Запрос POST /api/items с { id: 1000001 }]
    ↓
[Сервер проверяет уникальность, добавляет в allItems]
    ↓
[201 Created]
    ↓
[Ничего не делаем - UI уже правильный]
```

#### H. Batch-отправка (каждую секунду)

```
[Таймер setInterval(1000) срабатывает]
    ↓
[Проверка: queue.length > 0?]
    ↓
[Дедупликация на клиенте]
    ↓
    ├─ Группируем действия по ID
    ├─ select + deselect для одного ID = удаляем оба
    └─ Несколько reorder = оставляем последний
    ↓
[POST /api/batch с финальным массивом действий]
    ↓
[Сервер получает батч]
    ↓
[Серверная дедупликация (защита)]
    ↓
[Сервер применяет действия транзакционно]
    ↓
[200 OK]
    ↓
[Очищаем очередь]
```

#### I. Обработка ошибок и откат

```
[POST /api/batch вернул 500/400]

    ↓

```

```
    Пользователь  │  [Клик] → [Оптимистичный UI] → [Очередь]
```

### Ключевые принципы потоков данных

1. **UI сначала** — все изменения видны мгновенно
2. **Сервер — истина** — но мы доверяем, пока не получим ошибку
3. **Очереди спасают** — от тысяч запросов и потери данных
4. **Кэш ускоряет** — скролл вверх не требует запросов
5. **fullOrder правит** — DnD работает с полным порядком, пагинация с фильтрацией

### УРОВЕНЬ 1: Фронтенд-валидация (мгновенная, без запросов к серверу)

```typescript
// constants/validation.ts
export const ID_RANGE = {
  MIN_ID: 1,
  INITIAL_MAX_ID: 1_000_000, // Сгенерированные при старте
}

export const validateNewId = (idStr: string, currentMaxId: number): { valid: boolean; error?: string } => {
  if (!idStr.trim()) {
    return { valid: false, error: 'ID is required' }
  }
  
  const id = Number(idStr)
  
  if (isNaN(id)) {
    return { valid: false, error: 'ID must be a number' }
  }
  
  if (!Number.isInteger(id)) {
    return { valid: false, error: 'ID must be an integer' }
  }
  
  if (id < ID_RANGE.MIN_ID) {
    return { valid: false, error: `ID must be at least ${ID_RANGE.MIN_ID}` }
  }
  
  // Критическая проверка: ID не должен быть в диапазоне уже существующих
  if (id <= currentMaxId) {
    return { 
      valid: false, 
      error: `ID ${id} already exists (IDs 1-${currentMaxId.toLocaleString()} are taken)` 
    }
  }
  
  return { valid: true }
}
```

```jsx
// components/AddItemForm.tsx
const AddItemForm = ({ onAdd }) => {
  const [newId, setNewId] = useState('')
  const [error, setError] = useState('')
  const [touched, setTouched] = useState(false)
  
  // Получаем текущий максимальный ID из кэша/состояния
  const currentMaxId = useCurrentMaxId() // Хук, который следит за allItems
  
  const handleSubmit = (e) => {
    e.preventDefault()
    setTouched(true)
    
    const validation = validateNewId(newId, currentMaxId)
    
    if (!validation.valid) {
      setError(validation.error!)
      return
    }
    
    // Только если валидация ПРОЙДЕНА - отправляем на сервер
    onAdd(parseInt(newId))
    setNewId('')
    setTouched(false)
    setError('')
  }
  
  return (
    <form onSubmit={handleSubmit}>
      <input
        type="text"
        value={newId}
        onChange={(e) => {
          setNewId(e.target.value)
          if (touched) {
            const validation = validateNewId(e.target.value, currentMaxId)
            setError(validation.error || '')
          }
        }}
        onBlur={() => {
          setTouched(true)
          const validation = validateNewId(newId, currentMaxId)
          setError(validation.error || '')
        }}
        placeholder={`Enter new ID > ${currentMaxId.toLocaleString()}`}
      />
      <button type="submit" disabled={!!error || !newId}>
        Add Item
      </button>
      {error && <div className="error">⚠️ {error}</div>}
      <small>💡 IDs 1-{currentMaxId.toLocaleString()} already exist</small>
    </form>
  )
}
```

### УРОВЕНЬ 2: Серверная валидация (финальный арбитр)

```javascript
// server/routes/items.js
app.post('/api/items', (req, res) => {
  const { id } = req.body
  
  // 1. Базовая валидация
  if (!id || typeof id !== 'number' || !Number.isInteger(id)) {
    return res.status(400).json({ 
      error: 'ID must be an integer',
      code: 'INVALID_TYPE'
    })
  }
  
  if (id < 1) {
    return res.status(400).json({ 
      error: 'ID must be positive',
      code: 'NEGATIVE_ID'
    })
  }
  
  // 2. Проверка существования (включая сгенерированные и ранее добавленные)
  if (globalState.allItems.has(id)) {
    return res.status(409).json({ 
      error: `Item with ID ${id} already exists`,
      code: 'DUPLICATE_ID',
      existingId: id
    })
  }
  
  // 3. Добавление элемента
  const newItem = { id, createdAt: Date.now() }
  globalState.allItems.set(id, newItem)
  
  // 4. Успех
  res.status(201).json(newItem)
})
```

### Обработка ошибок на фронтенде

```jsx
// hooks/useAddItem.ts
const useAddItem = () => {
  const queryClient = useQueryClient()
  
  return useMutation({
    mutationFn: (id) => api.addItem(id),
    
    onMutate: async (id) => {
      // Оптимистичное обновление
      await queryClient.cancelQueries({ queryKey: ['left-items'] })
      const previousData = queryClient.getQueryData(['left-items', ''])
      
      queryClient.setQueryData(['left-items', ''], (old) => {
        if (!old) return old
        
        // Добавляем новый элемент в левый список
        const newPages = [...old.pages]
        const lastPage = newPages[newPages.length - 1]
        newPages[newPages.length - 1] = {
          ...lastPage,
          items: [...lastPage.items, { id }]
        }
        
        return { ...old, pages: newPages }
      })
      
      return { previousData }
    },
    
    onError: (error, id, context) => {
      // Откатываем оптимистичное обновление
      queryClient.setQueryData(['left-items', ''], context.previousData)
      
      // Показываем понятную ошибку пользователю
      if (error.response?.status === 409) {
        toast.error(`Item ${id} already exists on server`)
      } else {
        toast.error(`Failed to add item ${id}`)
      }
    },
    
    onSuccess: (newItem) => {
      toast.success(`Item ${newItem.id} added successfully`)
    }
  })
}
```

```
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                      СЕРВЕР (УРОВЕНЬ 2)                            │

│  │ ✅ Число?                                                   │ │

│  • Оставляем элемент в списке                                    │

│  • currentMaxId обновляется до 1000001                           │

```

### Что происходит при ошибке?

```jsx
// Если сервер вернул 409 Conflict (например, ID уже существует)
// Это может случиться, если:
// 1. Два пользователя одновременно добавили одинаковый ID
// 2. Мы не обновили currentMaxId (редко, но возможно)

onError: (error) => {
  // Откатываем UI
  queryClient.setQueryData(['left-items', ''], context.previousData)
  
  // Сообщаем пользователю
  toast.error(`Item ${id} already exists`)
  
  // Обновляем currentMaxId, чтобы предотвратить повторную попытку
  updateCurrentMaxId(id)
}
```

### Преимущества подхода

| Аспект | Без валидации | С валидацией |
|--------|---------------|--------------|
| **Запросы к серверу для ID ≤ 1M** | 1000 запросов | 0 запросов 🎉 |
| **Скорость обратной связи** | 100-300мс | Мгновенно |
| **Нагрузка на сервер** | Высокая | Низкая |
| **Пользовательский опыт** | "Почему не добавляется?" | Понятные ошибки сразу |

---

### Итоговое правило

> **Фронтенд НЕ отправляет на сервер ID, которые гарантированно не пройдут валидацию (≤ текущего максимального ID). Сервер остается финальным арбитром для race conditions и граничных случаев.**
